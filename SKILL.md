# Skill：课件 PDF 批量转 Obsidian 课程笔记

## 使用场景

当用户要求处理课程课件 PDF、批量整理 PDF、生成课程笔记、生成 Obsidian Markdown 笔记、生成学习自测题时，使用本 Skill。

用户通常会把 PDF 文件放在当前工作目录、`待处理PDF/` 文件夹，或用户明确指定的其他目录中。

本 Skill 的目标不是生成摘要，而是将课件内容完整重构为适合学习、复习、长期保存的课程笔记。

## 核心目标

将每一个 PDF 课件分别转换为一份标准化 Markdown 课程笔记。

处理结果应同时满足：

- 保留原课件中的全部实质知识点；
- 不改变原意；
- 不随意增加课件外内容；
- 去除课件式冗余；
- 避免 PPT 式碎片化表达；
- 避免将密集信息强行压缩为难读的长段落；
- 在段落、列表、表格和加粗小标题之间取得平衡；
- 形成适合 Obsidian 保存、检索、复习的 Markdown 笔记；
- 每个 PDF 输出一个同名 Markdown 文件；
- 每份笔记末尾生成带 Obsidian 原文链接的学习自测。

## 输入与输出

### 输入

优先处理用户明确指定的 PDF 文件或 PDF 文件夹。

如果用户没有明确指定，则按以下顺序查找：

1. 当前工作目录中的 `待处理PDF/` 文件夹；
2. 当前工作目录中的所有 PDF 文件。

用户会自行筛选 PDF，因此不要判断文件是否为课程课件。

### 输出

如果用户指定了输出目录，则输出到用户指定目录。

如果用户没有指定输出目录，则输出到当前工作目录中的：

`输出笔记/`

如果该文件夹不存在，应创建。

每个 PDF 输出一个同名 Markdown 文件。

同时生成：

`处理说明.md`

如果输出目录为 `输出笔记/`，则处理说明文件路径为：

`输出笔记/处理说明.md`

## 输出文件命名规则

每个 PDF 对应一个同名 Markdown 文件。

要求：

- 保留 PDF 主文件名；
- 去除 `.pdf` 后缀；
- 添加 `.md` 后缀；
- 不要将多个 PDF 合并为一个 Markdown；
- 不要将一个 PDF 拆成多个 Markdown，除非用户明确要求。

## 处理说明文件要求

`处理说明.md` 必须包括：

1. 本次处理的 PDF 文件清单；
2. 每个 PDF 的页数；
3. 每个 PDF 对应生成的 Markdown 文件名；
4. 是否存在无法提取、无法识别、疑似缺失的页面；
5. 是否存在图片、表格、流程图、示意图无法完全转写；
6. 需要人工核对的位置；
7. 每个输出 Markdown 是否已完成学习自测；
8. 每个输出 Markdown 是否已检查 Obsidian 链接格式。

## PDF 内容提取要求

对每一个 PDF，必须尽可能完整提取全文内容。

必须保留：

- 章节结构；
- 课程目录；
- 课程目标；
- 知识地图；
- 定义；
- 分类；
- 原理；
- 背景；
- 作用；
- 流程；
- 表格；
- 图示说明；
- 操作要求；
- 安全要求；
- 管理要求；
- 技术参数；
- 设备名称；
- 系统名称；
- 工器具名称；
- 工程案例；
- 项目案例；
- 典型场景；
- 结论性表述；
- 注意事项。

可以去除：

- 页码；
- 页眉；
- 页脚；
- 版权声明；
- 重复目录页；
- 装饰性文字；
- 无教学意义的过渡页；
- 完全重复的句子；
- PPT 中为了演示节奏拆开的重复表达。

不允许去除：

- 课程知识点；
- 分类标准；
- 流程步骤；
- 操作规范；
- 安全风险提示；
- 管理要求；
- 技术参数；
- 案例中的实质信息；
- 图表中的实质内容；
- 课件中的关键结论。

如果 PDF 中存在图片、流程图、表格、结构图或示意图：

1. 优先将其内容转写为文字说明；
2. 如果能识别大致含义，应整理为笔记内容；
3. 如果无法可靠识别，应在对应位置标注：

> 【此处原课件包含图示/表格，内容需人工核对】

并在 `处理说明.md` 中记录对应文件和页码。

## 课程结构重构要求

优先依据课件原有结构整理，包括：

- 目录；
- 知识地图；
- 章节标题；
- 模块标题；
- “一、二、三”等主标题；
- 原课件中明显的一级模块。

如果课件结构清晰，应尊重原结构。

如果课件结构松散，可以根据知识逻辑重新组织，但不能改变原意，也不能删减内容。

不要机械地把每一页 PPT 转换成一个标题。应合并同类内容，形成真正可学习、可复习的课程笔记。

## 标题层级规则

### 不要把课程名作为一级标题

课程名称由文件名体现，不需要在正文开头写成一级标题。

除非课程名称本身就是原课件中的一级模块标题，否则不要将课程名称写成：

`# 课程名称`

### 一级标题

一级标题 `#` 用于课程正文中的一级模块。

一级标题通常对应：

- 课件目录中的一级章节；
- 课程知识地图中的一级模块；
- 原课件中“一、二、三”级别的内容；
- 明显统领后续内容的大模块。

是否保留“一、二、三”等编号，应根据原课件目录风格决定。不要强行统一添加编号，也不要随意删除原有编号。

### 二级标题

二级标题 `##` 用于一级模块下的重要知识点、主要内容组或稳定的知识单元。

二级标题应简洁、明确，适合 Obsidian 链接引用。

### 三级标题

三级标题 `###` 用于局部内容分组。

三级标题不要过多。只有当内容确实需要形成独立小节时才使用。

### 禁止四级及以下标题

不要使用四级及以下标题：

`####`

`#####`

`######`

如果内容还需要继续分层，使用加粗小标题、列表或表格承接。

### 加粗小标题的使用

当某一内容需要独立呈现，但不适合升级为正式标题时，使用加粗小标题。

适合使用加粗小标题的情况包括：

- 典型工程；
- 典型案例；
- 项目示例；
- 单个设备说明；
- 局部技术特点；
- 某一操作要求；
- 某一注意事项；
- 某一风险点；
- 某一管理措施。

格式要求：

**名称或要点：**  
后接 1—3 句说明，必要时可继续使用紧凑列表。

## 正文开头规则

正文不需要用课程名作为开头。

如果原课件中有课程目标、学习目标、课程知识结构，可以在正文前部整理，但不要强行写成固定模板。

推荐使用加粗小标题或二级标题承接，具体取决于原课件结构。

如果这些内容不是课程一级模块，不要强行设置为一级标题。

## 正文组织原则

正文应按照课程一级模块展开。

每个模块应根据内容性质灵活组织，不要机械套模板。

允许使用以下形式：

- 连续段落；
- 紧凑列表；
- 有序列表；
- 表格；
- 加粗小标题；
- 必要的引用标注。

不同形式的使用原则如下：

- 段落用于解释原理、背景、意义、逻辑关系和连续性说明；
- 列表用于承载多个并列知识点、要求、对象、措施、案例或参数项；
- 有序列表用于流程、步骤、阶段、操作顺序；
- 表格用于分类、对比、字段稳定的信息；
- 加粗小标题用于独立但不宜升为正式标题的内容块。

不要把“避免空洞分点”理解为“完全不用分点”。  
也不要为了形式整齐而把所有内容都做成列表。

## 分点分条规则

### 应使用分点的情况

以下内容通常适合使用分点或列表：

- 多个并列知识点；
- 多个设备名称；
- 多个系统名称；
- 多个工程项目；
- 多个典型案例；
- 多个工器具；
- 多个技术路线；
- 多个参数项；
- 多个风险点；
- 多个控制措施；
- 多个操作要求；
- 多个注意事项；
- 多个适用场景；
- 多个优缺点；
- 多个影响因素。

如果这些信息全部压缩成一个长段落会降低可读性，应使用分点。

### 不应过度分点的情况

以下内容不宜拆成大量短分点：

- 连续性理论说明；
- 背景意义；
- 原理解释；
- 逻辑推导；
- 需要完整语境才能理解的内容；
- 只有少量简单并列项的内容；
- 原课件中本身就是完整段落的内容。

### 列表项质量要求

每个列表项应尽量包含完整信息。

避免只有孤立名词、孤立动词或过短碎片。

如果列表项只是单个名称，并且数量较少，可以改为一行紧凑表达。

如果列表项数量较多，或每项包含参数、状态、特征、作用、地点、工程意义等信息，应使用列表或表格。

## 典型工程、典型案例和项目示例规则

当课件中出现典型工程、典型案例、项目示例、应用案例、示范工程等内容时，不要将多个案例强行压缩成连续长段落。

如果每个工程、案例或项目都包含独立名称、技术特点、建设意义、关键参数、应用背景或实施效果，应让每个对象独立成块。

默认优先使用：

**工程 / 案例 / 项目名称：**  
后接简洁说明，说明内容应覆盖课件强调的定位、参数、技术特点、建设意义或应用效果。

如果数量较多且每项内容较短，可以使用紧凑列表。

如果字段稳定，可以使用表格。

不要默认把每个工程、案例或项目都设置为三级标题。只有在该对象内容较长、需要展开多个层次，或原课件本身就是这样分层时，才使用三级标题。

必须保留工程、案例或项目中的关键名称、参数、技术特征和课件强调的意义。

## 参数密集内容处理规则

当内容包含大量参数、等级、容量、距离、数量、时间、地点、对象名称等信息时，应优先考虑可读性。

处理原则：

- 参数较少且属于同一对象时，可放入段落；
- 多个对象各自带有参数时，优先使用列表；
- 多个对象具有相同字段时，优先使用表格；
- 不要把大量“名称 + 参数”连续堆在一个长句中；
- 不要删减参数；
- 不要随意改写参数单位；
- 不要擅自补全课件未提供的参数。

## 表格使用规范

适合用表格的内容包括：

- 分类对比；
- 技术路线对比；
- 工程项目对比；
- 设备项目对比；
- 流程阶段；
- 工器具用途；
- 岗位职责；
- 风险点与控制措施；
- 操作步骤与注意事项；
- 检查项目与判断标准；
- 对象、参数、特点之间具有稳定字段的信息。

不适合用表格的内容包括：

- 连续性理论说明；
- 背景意义；
- 概念解释；
- 需要完整语境才能理解的案例过程；
- 字段不稳定、强行制表反而降低可读性的内容。

表格不要过宽。

如果表格列数过多，应拆成多个较小表格，或者改为加粗小标题加段落说明。

不要为了形式统一而滥用表格。

## 流程和步骤处理规则

流程、步骤、阶段、操作顺序类内容应优先使用有序列表。

每一步应包含必要说明，不能只有一个短词。

如果流程中每一步还包含责任主体、操作内容、注意事项或输出结果，可以使用表格。

流程类内容必须保持原有顺序，不得随意调整。

## 概念和原理处理规则

定义、概念、原理、作用、意义类内容应优先使用段落式表达。

如果概念内部包含多个要点，可以在段落后使用紧凑列表补充。

不要把概念解释拆成过多孤立短句。

不要过度扩写课件没有提供的理论背景。

## 分类内容处理规则

分类内容应根据复杂程度选择呈现方式：

- 分类项较少、说明较短时，可使用紧凑列表；
- 分类项较多、字段稳定时，优先使用表格；
- 分类项需要较多解释时，可使用加粗小标题逐项说明；
- 分类内容涉及对比关系时，优先使用表格。

分类标准、分类名称和适用条件必须保留。

## 语言风格要求

使用正式、清晰、适合学习复习的中文。

语言应比课件更连贯，但不能改变课件原意。

不要使用口号化、营销化、过度抒情的表达。

不要使用过度口语化、短视频化、网感化表达。

不要出现以下表达：

- 一句话理解
- 一句话总结
- 简单来说
- 你可以把它理解为
- 本质上就是
- 这就像……
- 重点来了
- 超级重要
- 秒懂
- 保姆级
- 小白也能懂
- 直接拿捏
- 这个很关键
- 这一点必须记住

不要写成问答体正文。

不要写成思维导图。

不要使用 Mermaid。

不要为了追求简洁而删减知识点。

不要过度扩写原课件没有的内容。

可以对课件中表达不完整、衔接不自然的地方做轻度整理，但必须保持原意。

## 内容完整性要求

整理时必须优先保证内容完整。

不允许为了笔记更短而删减实质内容。

不允许将多个不同知识点模糊合并。

不允许删除课件中看似零散但具有复习价值的信息。

允许合并：

- 重复表达；
- 同一知识点在多页 PPT 中拆开的内容；
- 无教学意义的演示性过渡内容；
- 重复出现的标题页或目录页。

不允许合并到失去辨识度：

- 不同工程案例；
- 不同设备类型；
- 不同检修类别；
- 不同技术路线；
- 不同流程阶段；
- 不同安全措施；
- 不同管理要求。

## Obsidian 适配要求

正文标题应尽量稳定、简洁，方便内部链接。

避免出现大量重复标题。

如果原课件中存在多个同名内容，应适当增加限定词，使标题可区分。

Obsidian 链接中使用的文件名必须与当前 Markdown 文件名一致，不包含 `.md` 后缀。

不要在正文中加入不必要的 HTML、复杂插件语法或非通用 Markdown 语法。

## 学习自测要求

每份课程笔记末尾必须增加：

# 学习自测

如果原课件含有课后练习，可以直接使用原题目，再新增没有的题型。学习自测用于复习，不用于替代正文内容。

题型包括：

- 单选题；
- 多选题；
- 判断题；
- 简答题。

题量根据课程内容自动生成：

- 内容较短的 PDF：8—12 题；
- 内容中等的 PDF：12—20 题；
- 内容较长的 PDF：20—30 题。

不要生成过多题目，避免笔记末尾过重。

## 自测题覆盖范围

自测题应覆盖正文主要知识点，包括：

- 重要定义；
- 核心分类；
- 基本原理；
- 关键参数；
- 工作流程；
- 安全要求；
- 管理要求；
- 设备或工器具；
- 典型工程；
- 典型案例；
- 易混淆概念。

不要只考最表层的记忆点。

不要脱离正文内容出题。

不要加入课件之外的新知识作为考点。

## 自测题链接要求

每一道题后面都要提供 Obsidian 内部链接，链接到正文中对应知识点所在标题。

链接格式：

关联原文：[[文件名#标题名]]

要求：

- 文件名必须与当前 Markdown 文件名一致；
- 文件名不包含 `.md` 后缀；
- 标题名必须是正文中真实存在的一级、二级或三级标题；
- 不要链接到不存在的标题；
- 不要链接到学习自测自身；
- 如果某题涉及多个知识点，优先链接到最核心的正文标题。

在生成学习自测前，应先确认正文中可用的标题结构，再生成题目链接。

## 自测题格式

学习自测内部最多使用二级标题。

统一使用以下结构：

## 一、单选题

题目应包含题干、选项、答案、解析和关联原文。

## 二、多选题

题目应包含题干、选项、答案、解析和关联原文。

## 三、判断题

题目应包含题干、答案、解析和关联原文。

## 四、简答题

题目应包含题干、参考答案和关联原文。

不要在学习自测中使用三级以下过细结构。

## 自查要求

每处理完一个 PDF 后，必须检查：

1. 是否生成了同名 Markdown 文件；
2. 是否没有将课程名错误设置为正文一级标题；
3. 一级标题是否主要对应课程一级模块；
4. 是否没有使用四级及以下标题；
5. 原课件主要章节是否全部覆盖；
6. 定义、分类、流程、设备、工具、参数、案例是否保留；
7. 是否误删了看似重复但实际有教学意义的内容；
8. 是否存在无法识别的图表、图片或页面；
9. 是否出现大量空洞分点；
10. 是否出现过度段落化、密集信息难以阅读的问题；
11. 是否对典型工程、典型案例、项目示例进行了独立呈现；
